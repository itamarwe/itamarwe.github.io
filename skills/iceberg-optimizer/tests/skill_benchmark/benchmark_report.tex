\section{Executive Summary}

The \textbf{iceberg-optimizer} skill is a Claude Code skill that diagnoses
Apache Iceberg tables and produces a ranked, cost-aware maintenance plan.
This report documents two benchmark runs.  In the first run, every scenario
passed the LLM judge evaluation with an average judge score of \textbf{4.8/5};
a partition-misalignment scenario scored 4/5 because the skill recommended
z-order compaction as the primary fix rather than partition evolution.  After
targeted improvements to the skill (partition evolution rank fix, three-mode
execution architecture, and book-derived reference additions), a second run
achieved a perfect \textbf{5.0/5} --- all five scenarios score 5/5.

The skill correctly handles cold archives (recommending near-nothing), streaming
thin-spread tables (bin-pack + z-order + write-time tuning), GDPR compliance
tables (physical-deletion sequence with snapshot-history warning),
partition-misalignment (partition evolution promoted above compaction), and
snapshot-only bloat (expiry without touching data files).

A parallel skill review (Section~\ref{sec:skill-review}) identifies further
improvements to the late-data loop and no-query-log fallback path.

\section{Skill Architecture}

\subsection{Design Principle}

The skill follows one governing rule:
\emph{observe before you ask, ask before you decide, simulate before you
recommend.}  It reads everything it can from table metadata and query logs
before asking the user a single question, then only asks about intent that
metadata cannot reveal.

\subsection{Three-Mode Execution Model}

Phase~0 detects one of three execution modes before any other work begins:

\begin{description}[leftmargin=2.5em,labelwidth=2em]
  \item[Direct]   All required inputs (profile, workload, simulator output) are
        provided on the command line.  The skill proceeds immediately to
        Phase~1 with no user interaction.
  \item[Ask-User] Some inputs are absent.  The skill asks targeted questions to
        fill the gaps, then proceeds.
  \item[Exported] The invocation references a pre-exported snapshot bundle.
        The skill reads the bundle and proceeds without running scripts live.
\end{description}

In addition, the skill now loads reference files \emph{gradually}: only the
sections relevant to the current phase are loaded rather than all files upfront.
At Phase~5, \code{procedures.md} is grepped for only the chosen engine and
selected actions, keeping the active context window narrow and the plan output
focused.

\subsection{Phase Flow}

\begin{table}[ht]
\centering
\caption{Iceberg optimizer phase flow}
\begin{tabular}{clp{9cm}}
\toprule
\textbf{Phase} & \textbf{Name} & \textbf{Purpose} \\
\midrule
0  & Scope \& Safety      & Identify table, engine, catalog; read-only mode until Phase~5. \\
1  & Profile              & Run \code{profile\_table.py} against \code{\$files}/\code{\$snapshots}/\code{\$manifests}: file sizes, delete pressure, snapshot bloat, partition fan-out. \\
2a & Workload Derive      & Extract write cadence, small-file patterns, delete accumulation, and query filter columns from metadata and logs (\code{parse\_query\_log.py}). \\
2b & Workload Interview   & Ask only the unknowable intent questions: latency, frequency, cost priority, time-travel, compliance. \\
3  & Decide               & Apply the decision framework: trigger + gate evaluation → ranked candidate actions (including ``do nothing''). \\
4  & Simulate             & Run \code{simulate.py} to compare Do-nothing / Light / Targeted-sort / Aggressive / Storage-min on four cost axes. \\
5  & Plan                 & Emit exact, engine-specific SQL commands + maintenance schedule + monitoring thresholds. \\
\bottomrule
\end{tabular}
\end{table}

\subsection{Candidate Actions}

The decision framework defines twelve candidate actions (A--L and Z):

\begin{description}[leftmargin=2.5em,labelwidth=2em]
  \item[A]  Bin-pack compaction --- resize small files.
  \item[B]  Sort compaction --- cluster by a single key.
  \item[C]  Z-order compaction --- multi-dimensional clustering (2--4 high-cardinality cols).
  \item[D]  Partition evolution --- zero-downtime re-partitioning via metadata.
  \item[E1] Equality-delete compaction --- urgent; penalises every scan; GDPR path bypasses query-frequency gate.
  \item[E2] Position-delete compaction --- lower urgency; row-level seek per file.
  \item[F]  Snapshot expiry --- metadata-only; always include; size to intent.
  \item[G]  Manifest rewrite / clustering --- consolidate or cluster manifests for planning efficiency.
  \item[H]  Orphan-file removal --- destructive; dry-run first; after snapshot expiry.
  \item[I]  Bloom filters --- high-cardinality equality lookups only.
  \item[J]  Write-time tuning --- fix the ingestion before compacting its output.
  \item[K]  Write-time sort order --- free clustering for future writes; no rewrite cost.
  \item[L]  Format-version upgrade --- one-time prerequisite for row-level delete workloads.
  \item[Z]  Do nothing / minimal --- explicit outcome for cold, rarely-read tables.
\end{description}

\section{Benchmark Methodology}
\label{sec:methodology}

\subsection{Approach}

A \emph{skill-level} benchmark evaluates the full end-to-end recommendation
rather than any individual script.  It answers: \emph{given a table's complete
profile and scripted interview answers, does the skill give the right advice?}

The harness (\code{run\_benchmark.py}) works as follows:
\begin{enumerate}
  \item Load \code{SKILL.md} and all \code{references/*.md} files as the system
        prompt --- the same context Claude receives when the skill is invoked.
  \item For each scenario, build a single-turn user message containing:
    \begin{itemize}
      \item Pre-computed \code{profile.json} (from \code{profile\_table.py}).
      \item Pre-computed \code{workload.json} (from \code{parse\_query\_log.py}).
      \item Pre-computed simulator output (from \code{simulate.py}).
      \item Scripted Phase~2b interview answers from \code{scenarios.json}.
    \end{itemize}
  \item Call the \code{claude~-p} CLI in non-interactive mode (no tool use,
        pure text response).
  \item Run \textbf{LLM-as-judge}: a separate Claude invocation is given the
        scenario's \code{expected\_outcome} description and the skill's response;
        it returns a score 1--5 and a binary PASS/FAIL (threshold: score~$\geq 3$).
\end{enumerate}

The judge evaluates correctness of reasoning rather than surface-level term
presence.  Each scenario's \code{expected\_outcome} is written in plain English
(e.g.\ ``the correct answer is snapshot expiry only; recommending compaction
is incorrect'') so the judge can assess whether the skill understood the
problem, not just whether it produced specific keywords.

\subsection{Fixture Generation}

Fixtures are deterministic, generated by \code{generate\_fixtures.py} using
the same data builders as the pytest unit tests.  Each fixture represents a
physically distinct table state:

\begin{itemize}
  \item File sizes, record counts, and delete-file types are constructed
        explicitly (not loaded from disk).
  \item Snapshot histories are synthetic (weekly / 30-second / daily cadence
        as appropriate per scenario).
  \item Workload JSON is hand-crafted to match the scenario's query access
        pattern (e.g.\ \code{partition\_prune\_rate = 0.0} for
        partition\_misalignment).
\end{itemize}

\subsection{Second Benchmark Run --- Post-Improvement Results}

The first benchmark run (prior to the improvements described below) achieved an
average judge score of \textbf{4.8/5}, with partition\_misalignment scoring
4/5.  The second run, after the following three categories of changes, achieves
a perfect \textbf{5.0/5} --- all five scenarios score 5/5.

\begin{enumerate}

  \item \textbf{Partition evolution rank fix.}  An explicit rank override was
        added to \code{decision-framework.md}: when
        \code{partition\_prune\_rate < 0.2} AND the dominant filter column is
        not in the partition spec, promote Action~D (partition evolution) above
        B/C (sort/z-order).  This directly fixed the 4/5 on
        partition\_misalignment.

  \item \textbf{Architecture restructure (SKILL.md).}  Three-mode detection
        (Direct / Ask-User / Exported) at Phase~0.  Per-phase gradual context
        loading: only the reference sections needed at each phase are loaded
        rather than all files upfront.  At Phase~5, \code{procedures.md} is
        grepped for only the matching engine and chosen actions.

  \item \textbf{Book-derived additions.}  Twelve gaps from
        \emph{Architecting Apache Iceberg Lakehouse} were incorporated into the
        reference files, including: Iceberg~v3 deletion vectors,
        \code{rewrite\_position\_delete\_files} procedure, file sizing guidance,
        CDC time-based snapshot expiry, conditional compaction, delete-file
        sawtooth diagnostic, multi-engine flexibility note, maintenance failure
        patterns, ordering justification, IAM/RBAC principles, and format
        version migration notes.

\end{enumerate}

\section{Scenario Deep Dives}

\subsection{cold\_archive: ``Do Nothing'' Is a Valid Answer}

A cold archive with 300 MB files, no delete pressure, and 3--5 queries per year
is \emph{physically healthy} --- maintenance would cost more compute than it
saves.  The skill:
\begin{itemize}
  \item Showed a gate evaluation table listing every action with its verdict
        (``Skip'').
  \item Quantified why: at the real query rate ($\approx 0.4$ queries/month),
        targeted-sort costs \$0.13/month vs do-nothing's \$0.09/month.
  \item Emitted an explicit ``What you are NOT doing'' table to pre-empt
        second-guessing.
\end{itemize}

\subsection{streaming\_thin\_spread: All Four Actions Required}

The problem is multi-layered: 5\,000 small files exist \emph{today}
(compaction) AND ingestion writes new thin-spread files every 30 seconds
(write-time tuning, so the problem does not recur).  The skill produced the
full correct sequence:
\begin{enumerate}
  \item Bin-pack compaction of cold partitions only (do not fight the live writer).
  \item Z-order rewrite on \code{tenant\_id, event\_time}.
  \item \code{write.distribution-mode = hash} to prevent future thin-spread.
  \item Aggressive snapshot expiry (\code{retain $\approx$ 10}).
\end{enumerate}

\subsection{gdpr\_deletes: Compliance Path Bypasses the Query-Frequency Gate}

Equality-delete compaction is \emph{non-optional} regardless of query
frequency for a GDPR table.  A performance-only framework would skip
compaction for a table read daily but not interactively; the compliance gate
must override it.  The skill correctly:
\begin{itemize}
  \item Identified \code{eq\_delete\_pressure = 10\%} as the trigger.
  \item Applied the compliance path: frequency gate bypassed.
  \item Produced the correct sequence: \code{rewrite\_data\_files(remove-dangling-deletes=true)}
        $\to$ \code{expire\_snapshots(retain\_last=1)}.
  \item Warned that retained snapshot history exposes logically-deleted rows ---
        a GDPR liability.
\end{itemize}

\subsection{partition\_misalignment: Invisible Without the Workload}

The profile alone shows a \emph{healthy} table: all flags false, 256 MB
files, no deletes.  The problem only appears in the workload:

\begin{quote}
  \emph{\code{partition\_prune\_rate = 0.0} --- every query full-table scans
  12.5 GB because the partition key (\code{event\_date}) does not match the
  dominant filter (\code{tenant\_id}).  This is invisible to any profiler that
  reads only metadata.}
\end{quote}

In the second benchmark run the skill correctly identified the root cause as a
partition-column mismatch and promoted \textbf{partition evolution} (Action~D)
as the primary fix: \code{ALTER TABLE\ldots ADD PARTITION FIELD bucket(N,
tenant\_id)} is a metadata-only operation (zero data rewrite, zero downtime)
that eliminates full-table scans permanently.  Z-order compaction was listed as
a secondary alternative for immediate read improvement, with a clear explanation
that it requires a full data rewrite and only improves skipping \emph{within}
scans that still touch all partitions.  The judge awarded 5/5.

The first benchmark run scored 4/5 on this scenario because the skill surfaced
z-order as the primary action; the partition evolution rank fix in
\code{decision-framework.md} resolved this.

\subsection{snapshot\_bloat\_only: Don't Fix What Isn't Broken}

The 1\,200-snapshot table tests whether the skill avoids treating snapshot
bloat as a data-file problem.  Data files are at target size (256 MB, 5 files)
--- compaction would rewrite them for no gain.  The skill:
\begin{itemize}
  \item Identified 1\,200 snapshots as the sole problem signal.
  \item Recommended only \code{expire\_snapshots} with a 2-week retention window.
  \item Explicitly stated that data file compaction is not needed.
\end{itemize}

\section{Skill Review}
\label{sec:skill-review}

A qualitative review of all skill files was conducted alongside the benchmark.

\subsection{Strengths}

\begin{itemize}
  \item \textbf{Trigger/gate separation.}  The decision framework clearly
        distinguishes metadata triggers (``small files exist'') from intent
        gates (``queries frequent enough to amortise the rewrite'').  This
        prevents the classic over-engineering failure mode.
  \item \textbf{``Do nothing'' as a first-class outcome.}  Action Z is
        documented with the same level of detail as aggressive compaction.
        The simulator's Do-nothing baseline makes this concrete.
  \item \textbf{Compliance path.}  The GDPR sequence is correct, ordered, and
        explicitly overrides the query-frequency gate --- the most common mistake
        in Iceberg GDPR guidance is treating logical deletion as sufficient.
  \item \textbf{Derive-first interview.}  Phase~2a derives all answerable
        signals from metadata before asking the user anything.  This reduces
        interview fatigue and prevents asking questions already known.
  \item \textbf{Honest about heuristics.}  \code{scan\_fraction} is explicitly
        called a heuristic in SKILL.md, with instructions for replacing it with
        measured numbers.
\end{itemize}

\subsection{Top 3 Recommended Improvements}

\begin{enumerate}

\item \textbf{[High impact] Elevate partition evolution for zero-prune scenarios.}

  Sort/Z-order (B/C) currently ranks above Partition evolution (D).  This
  inverts when \code{partition\_prune\_rate = 0.0} and the top filter column
  is not in the partition spec: partition evolution is a metadata-only operation
  (zero data rewrite) that eliminates full-table scans permanently, while
  z-order is a full data rewrite that only improves skipping within a scan
  that still touches all partitions.

  Recommended addition to Action~D:
  \begin{quote}
  \textit{If \code{partition\_prune\_rate < 0.2} AND the dominant filter column
  is not in the current partition spec, promote D above B/C in ROI ranking.}
  \end{quote}

  (\emph{Note: this fix was applied to \code{decision-framework.md} after the
  benchmark run.})

\item \textbf{[Medium impact] Close the late-data loop to actionable gates.}

  The skill correctly derives \code{late\_data = true} from snapshot summaries
  but does not connect this to a decision gate.  Late-arriving data breaks the
  ``compact only cold partitions'' assumption: late writes land in partitions
  that appear cold, causing conflicts with the compaction job.

  Recommended additions: add ``AND \code{late\_data $\neq$ true}'' to the
  Action~B sort-compaction gate; add a scheduling note to widen the cold-partition
  window by the observed maximum late-arrival lag.

\item \textbf{[Medium impact] Provide a ``no query logs'' fallback in Phase~2.}

  SKILL.md describes two ways to obtain measured bytes-scanned (Spark event
  log, Trino queries) but does not have an explicit fallback for when neither
  is available.  The cold-start path should be documented:

  \begin{quote}
  \textit{If no query logs are available, ask the user to run
  \code{EXPLAIN ANALYZE} on 3--5 representative queries and pass the output
  via \code{--explain-analyze}.  If even that is unavailable, the simulator
  estimates from \code{total\_gb $\times$ (1 - prune\_rate)} --- label this
  clearly as a heuristic.}
  \end{quote}

\end{enumerate}

\section{Conclusion}

The iceberg-optimizer skill achieves a perfect \textbf{5.0/5} across all five
benchmark scenarios in the second run (up from 4.8/5 in the first run).  Key
findings:

\begin{enumerate}
  \item \textbf{The ``do nothing'' path works.}  The skill correctly chose
        minimal action for the cold archive and explicitly quantified why
        maintenance would be net-negative.
  \item \textbf{The GDPR compliance path is correct.}  The non-optional
        compaction + snapshot-expiry sequence was produced without prompting,
        and the snapshot-history liability was called out.
  \item \textbf{Profile alone is not enough.}  The partition-misalignment
        scenario has a healthy profile; the problem is only visible in the
        workload.  The skill correctly diagnosed it from the workload signal.
  \item \textbf{Partition evolution is now correctly ranked.}  The rank override
        added to \code{decision-framework.md} ensures that when
        \code{partition\_prune\_rate < 0.2} and the dominant filter column is
        not a partition key, partition evolution is promoted above sort/z-order
        compaction.  This lifted partition\_misalignment from 4/5 to 5/5.
  \item \textbf{Architecture improvements reduce hallucination surface.}  The
        three-mode detection model and gradual context loading keep the active
        context focused on the current phase and engine, improving plan
        precision across all scenarios.
\end{enumerate}
